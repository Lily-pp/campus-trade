# CampusTrade 开发指南

## 1. 文档目的

本文档用于指导项目成员进行 CampusTrade 后续功能开发。

主要目标：

-   统一项目开发流程；
-   规范数据库、后端、前端修改方式；
-   降低多人协作产生的冲突；
-   保证新增功能可以稳定集成。

适用于：

-   新增业务模块；
-   修改已有功能；
-   数据库结构调整；
-   Git 分支协作。

# 5. 数据库开发规范

## 5.1 Migration 规范

数据库文件位置：

    database/migrations

当前版本：

    V001 ~ V011

历史 migration 不允许修改。

例如：

不要：

    修改 V005_cleanup_and_seed.sql

应该新增：

    V012_new_feature.sql

------------------------------------------------------------------------

## 5.2 新增表流程

例如新增校园论坛：

创建：

    database/migrations/V012_forum.sql

内容：

``` sql
CREATE TABLE forum_posts(
    id SERIAL PRIMARY KEY,
    user_id INTEGER,
    content TEXT
);
```

要求：

-   使用新的版本号；
-   保持版本连续；
-   添加必要外键；
-   注意字段命名规范。

------------------------------------------------------------------------

## 5.3 数据库升级

新数据库：

执行：

``` bash
npm run db:init
```

已有数据库：

执行：

``` bash
npm run db:migrate
```

不要使用 `db:init` 覆盖已有业务数据库。

------------------------------------------------------------------------

# 6. 后端开发规范

后端目录：

    server/routes

每个业务模块建议独立文件。

例如：

商品：

    routes/items.js

订单：

    routes/orders.js

失物招领：

    routes/lostFound.js

------------------------------------------------------------------------

## 新增接口流程

例如新增论坛模块：

创建：

    server/routes/forum.js

定义：

``` javascript
router.get()

router.post()

router.put()

router.delete()
```

然后在：

    server/app.js

注册：

``` javascript
app.use(
 '/api/forum',
 forumRouter
)
```

------------------------------------------------------------------------

## 后端注意事项

新增接口时注意：

### 权限

检查：

-   用户身份；
-   管理员权限；
-   数据所有权。

### 数据验证

检查：

-   必填字段；
-   参数类型；
-   数据范围。

### 数据库操作

涉及：

-   创建订单；
-   修改库存；
-   状态变化；

应考虑事务。

    ↓

## 期望交易地点

数据库字段：

    expected_address

    expected_longitude

    expected_latitude

作用：

保存校园线下交易位置。

未来可以扩展：

-   附近商品；
-   距离排序；
-   地图展示。

------------------------------------------------------------------------

## 失物招领

采用独立模型：

    lost_posts
    found_items
    lost_found_points

不要混入：

    items

原因：

失物招领与商品交易业务不同。

# 9. Git 协作规范

## 分支规范

不要直接开发：

    merge-b-version

推荐：

    feature/功能名称

例如：

    feature/forum
    feature/search-optimize

------------------------------------------------------------------------

## 开发流程

    创建功能分支

    ↓

    开发

    ↓

    本地测试

    ↓

    提交 commit

    ↓

    Push

    ↓

    Pull Request

    ↓

    合并

------------------------------------------------------------------------

## Commit 建议

格式：

    type: description

例如：

功能：

    feat: add forum module

修复：

    fix: solve order status bug

文档：

    docs: update development guide

------------------------------------------------------------------------

# 10. 合并前检查

提交 PR 前：

## 后端

检查：

``` bash
node --check 文件名
```

------------------------------------------------------------------------

## 前端

检查：

-   页面正常加载；
-   路由正常；
-   API 请求正常。

------------------------------------------------------------------------

## 数据库

检查：

-   migration 可以执行；
-   字段存在；
-   外键正确。

------------------------------------------------------------------------

# 11. 常见开发问题

## 字段不存在

错误：

    column xxx does not exist

解决：

执行：

    npm run db:migrate

------------------------------------------------------------------------

## 合并冲突

处理：

1.  查看冲突文件；
2.  删除冲突标记：

```{=html}
<!-- -->
```
    <<<<<<<
    =======
    >>>>>>>

3.  测试功能；
4.  git add；
5.  commit。

# 13. 开发原则

后续开发遵循：

1.  不修改历史 migration；
2.  新功能独立模块化；
3.  数据库、后端、前端同步设计；
4.  合并前必须测试；
5.  保持代码结构清晰。

这样可以保证 CampusTrade 在持续增加功能时仍然保持稳定。
